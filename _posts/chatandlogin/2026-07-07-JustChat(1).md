TestFlight 심사를 기다리는 동안 새로운 미니 프로젝트를 하나 해보려고 한다.

프로젝트 이름은 JustChat. 소셜 로그인과 Firebase를 이용한 실시간 채팅 기능을 구현하는 앱이다.

이 주제를 고른 이유는 2년 전 분식집을 주제로 한 팀 프로젝트에서 이 부분(소셜 로그인, 채팅)을 구현할 때의 아쉬움 때문이다. 

당시엔 관련 지식과 경험이 부족해서 AI에게 구현을 거의 맡기다시피 했고, 일정도 빠듯해서 그 과정을 제대로 정리하지도 못했다. 그때 못 다한 이해와 정리를, 이번엔 시간에 쫓기지 않는 상태에서 제대로 해보고 싶다.

일단 UI의 경우 RunWay와 똑같이 AI에게 만들어 달라고 했다.

우선 첫화면은 다음과 같다.

![](https://pub-1fd8ca6711bd4f3f8b74d88a697b50f9.r2.dev/2026-07-06-JustChat1/homeview.png){: width="50%" height="50%"}

다만 소셜로그인 기능을 구현하기전에 먼저 전체적인 진행 순서를 간단하게 잡고 가려한다.

1. **Firebase 연동** - GoogleService-Info.plist 설정, SDK 연결, Auth/Firestore 콘솔 세팅
2. **소셜 로그인 연동** - Apple/Google Sign-In, async/await로 감싸기
3. **채팅 기능 구현** - Firestore 실시간 리스너, 메시지 송수신

모델링은 각 단계 진행하면서 필요할 때마다 그때그때 확정해나갈 생각이다. 처음부터 스키마를 다 잡고 시작하기보다, 실제로 붙여보면서 필요한 형태로 다듬는 쪽이 낫다고 판단했다.

## FirebaseApp 초기화

Firebase SDK를 추가했으면 앱 진입점에서 초기화를 해줘야 한다.

[Firebase Docs](https://firebase.google.com/docs/ios/setup?hl=ko){:target="_blank"}에서는 SwiftUI 앱이라도 `AppDelegate`를 만들어서 초기화하는 방식을 권장한다.

Firebase 내부적으로 `UIApplicationDelegate`의 콜백(푸시 알림, URL 스킴 처리 등)에 의존하는 부분이 있기 때문이다. 

`@UIApplicationDelegateAdaptor`는 SwiftUI 앱에서 `AppDelegate`를 다시 연결해주는 브릿지 역할을 한다.

```swift
class AppDelegate: NSObject, UIApplicationDelegate {
    func application(_ application: UIApplication,
                     didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey : Any]? = nil) -> Bool {
        FirebaseApp.configure()
        return true
    }
}

@main
struct JustChatApp: App {
    
    @UIApplicationDelegateAdaptor(AppDelegate.self) var delegate

    var body: some Scene {
        WindowGroup {
            RootView()
        }
    }
}
```

이렇게 기본작업은 끝났다.

### Apple Login

우선 Apple 로그인부터 해보려 한다.

Apple 로그인의 핵심은 **nonce**다. 전체 흐름은 이렇다.

1. 앱에서 임의의 nonce를 생성
2. nonce를 SHA256으로 해시해서 Apple에 전달
3. Apple이 인증 후 `identityToken`을 돌려줌
4. Firebase에 `identityToken` + 원본 nonce를 같이 넘겨서 검증

nonce가 필요한 이유는 토큰 재사용 공격을 막기 위해서다. Apple이 발급한 토큰이 중간에 탈취됐더라도, Firebase가 "이 토큰은 이 nonce로 요청한 것"임을 검증하기 때문에 다른 곳에서 재사용할 수 없다.

구현은 [Firebase Docs](https://firebase.google.com/docs/auth/ios/apple?hl=ko){:target="_blank"}를 기준으로 진행한다. Docs에서 제공하는 샘플 코드가 있기 때문에 흐름을 먼저 파악하고 코드를 하나씩 뜯어보는 방식으로 접근해보려 한다.

---

UI 쪽은 `AuthenticationServices`에서 제공하는 `SignInWithAppleButton`을 쓰면 된다. 

[Apple Guidelines](https://developer.apple.com/design/human-interface-guidelines/sign-in-with-apple){:target="_blank"}상 Sign in with Apple 버튼은 직접 커스텀하지 않고 이 컴포넌트를 쓰는 것을 권장한다. 첫 번째 인자로 `.signIn`, `.continue`, `.setUp` 중 하나를 넘겨서 버튼 텍스트를 결정한다.

```swift
import AuthenticationServices

SignInWithAppleButton(.continue) { request in
    request.requestedScopes = [.fullName, .email]
} onCompletion: { _ in
    appleTapped()
}
```

`onCompletion`에서 결과를 직접 처리하지 않고 `appleTapped()`로 넘기는 이유는 실제 인증 흐름을 `AppleLoginService`에서 담당하기 때문이다. UI는 트리거 역할만 한다.

#### 2. nonce 생성 함수 만들기

우선 `AppleLoginService`라는 클래스를 하나 만들어 주었다. 로그인 관련 로직을 한 곳에서 관리하기 위해서다.

nonce 생성 함수와 SHA256 해시 함수는 [Firebase Docs](https://firebase.google.com/docs/auth/ios/apple?hl=ko){:target="_blank"}에 샘플 코드가 있어서 그대로 가져왔다.

```swift
// AppleLoginService
import CryptoKit

private func randomNonceString(length: Int = 32) -> String {
  precondition(length > 0)
  var randomBytes = [UInt8](repeating: 0, count: length)
  let errorCode = SecRandomCopyBytes(kSecRandomDefault, randomBytes.count, &randomBytes)
  if errorCode != errSecSuccess {
    fatalError(
      "Unable to generate nonce. SecRandomCopyBytes failed with OSStatus \(errorCode)"
    )
  }

  let charset: [Character] =
    Array("0123456789ABCDEFGHIJKLMNOPQRSTUVXYZabcdefghijklmnopqrstuvwxyz-._")

  let nonce = randomBytes.map { byte in
    charset[Int(byte) % charset.count]
  }

  return String(nonce)
}

private func sha256(_ input: String) -> String {
  let inputData = Data(input.utf8)
  let hashedData = SHA256.hash(data: inputData)
  let hashString = hashedData.compactMap {
    String(format: "%02x", $0)
  }.joined()

  return hashString
}
```

두 함수의 역할이 다르다. `randomNonceString`은 원본 nonce를 만들고, `sha256`은 그걸 해시해서 Apple에 전달할 값으로 변환한다. Firebase 검증 단계에서 원본 nonce와 해시값을 대조하기 때문에 둘 다 필요하다.

#### 3. ASAuthorizationAppleIDRequest 설정

`ASAuthorizationAppleIDProvider`로 request를 만들고, 요청할 정보의 범위(scope)를 지정한다. 여기서 2번에서 만든 nonce를 해시해서 request에 담아 Apple에 전달한다.

이때 `ASAuthorizationControllerDelegate`와 `ASAuthorizationControllerPresentationContextProviding` 두 프로토콜을 채택해야 한다. request 설정 후 `authorizationController`에 delegate를 지정하는 부분이 그 연결점이다.

```swift
fileprivate var currentNonce: String?

func startSignInWithAppleFlow() {
    let nonce = randomNonceString()
    currentNonce = nonce
    let appleIDProvider = ASAuthorizationAppleIDProvider()
    let request = appleIDProvider.createRequest()
    request.requestedScopes = [.fullName, .email]
    request.nonce = sha256(nonce)

    let authorizationController = ASAuthorizationController(authorizationRequests: [request])
    authorizationController.delegate = self
    authorizationController.presentationContextProvider = self
    authorizationController.performRequests()
}
```

- `currentNonce` - 원본 nonce를 저장해두는 프로퍼티. 나중에 Firebase 검증 단계에서 다시 꺼내 쓴다.
- `requestedScopes` - Apple에 요청할 정보 범위. `.fullName`과 `.email`을 지정했다.
- `request.nonce` - 해시된 nonce를 Apple에 전달한다. 원본이 아닌 해시값을 넘기는 이유는 원본은 Firebase 검증용으로만 쓰기 위해서다.

#### 4. delegate 구현 (identityToken 추출 및 Firebase 로그인)

`ASAuthorizationControllerDelegate` 프로토콜은 extension으로 분리했다.
성공 케이스에서는 `appleIDCredential`에서 `identityToken`을 꺼내고, Firebase credential로 변환해서 로그인까지 한 번에 처리한다.

Docs 코드는 completion handler 방식이었는데 async/await로 바꿨다.

여기서 `authorizationController.presentationContextProvider = self` 때문에 `ASAuthorizationControllerPresentationContextProviding` 프로토콜도 함께 채택해야 한다. 이 프로토콜은 Apple 로그인 창을 어느 window에 띄울지 시스템에 알려주는 역할이다. 없으면 로그인 시트가 어디에 표시돼야 할지 몰라서 동작하지 않는다.

UIKit에서는 `view.window`로 바로 접근했는데, SwiftUI에서는 `UIApplication.shared.connectedScenes`를 통해 window를 가져와야 한다.

`connectedScenes`는 현재 앱에 연결된 scene 목록이다. iOS에서 scene은 앱의 UI 인스턴스 하나를 의미하는데, `compactMap({ $0 as? UIWindowScene })`으로 window를 가진 scene만 필터링하고 `.first?.windows.first`로 첫 번째 window를 꺼낸다. 멀티 윈도우를 지원하지 않는 일반적인 앱이라면 항상 하나만 존재한다.

```swift
extension AppleLoginService: ASAuthorizationControllerPresentationContextProviding {
    func presentationAnchor(for controller: ASAuthorizationController) -> ASPresentationAnchor {
        guard let window = UIApplication.shared.connectedScenes
            .compactMap({ $0 as? UIWindowScene })
            .first?.windows
            .first else {
            fatalError("No window found")
        }
        return window
    }
}

extension AppleLoginService: ASAuthorizationControllerDelegate {
    func authorizationController(controller: ASAuthorizationController, didCompleteWithAuthorization authorization: ASAuthorization) {
        if let appleIDCredential = authorization.credential as? ASAuthorizationAppleIDCredential {
            guard let nonce = currentNonce else {
                fatalError("Invalid state: A login callback was received, but no login request was sent.")
            }
            guard let appleIDToken = appleIDCredential.identityToken else {
                print("Unable to fetch identity token")
                return
            }
            guard let idTokenString = String(data: appleIDToken, encoding: .utf8) else {
                print("Unable to serialize token string from data: \(appleIDToken.debugDescription)")
                return
            }

            let credential = OAuthProvider.appleCredential(
                withIDToken: idTokenString,
                rawNonce: nonce,
                fullName: appleIDCredential.fullName
            )

            Task {
                do {
                    let authResult = try await Auth.auth().signIn(with: credential)
                    print("Firebase 로그인 성공: \(authResult.user.uid)")
                } catch {
                    print("Firebase 로그인 실패: \(error.localizedDescription)")
                }
            }
        }
    }

    func authorizationController(controller: ASAuthorizationController, didCompleteWithError error: Error) {
        print("Sign in with Apple errored: \(error)")
    }
}
```

`OAuthProvider.appleCredential`에 세 가지를 넘긴다.

- `withIDToken` - Apple이 발급한 identity token. Apple 사용자 인증의 증명이다.
- `rawNonce` - 원본 nonce. Firebase가 Apple에 넘겼던 해시값과 대조해서 토큰 유효성을 검증한다.
- `fullName` - 사용자 이름. Apple은 최초 로그인 시에만 이 값을 제공한다.

guard문을 단계적으로 쓰는 이유는 각 단계마다 실패 지점이 다르기 때문이다.

- `currentNonce`가 없으면 요청 자체가 없었던 것
- `identityToken`이 없으면 Apple 인증이 제대로 완료되지 않은 것
- `idTokenString` 변환 실패면 토큰 데이터가 손상된 것

`currentNonce`를 중간에 잃어버리면 Firebase 검증 단계에서 `.MissingOrInvalidNonce` 에러가 난다.

#### 5. 재인증 (Reauthentication)

Apple 토큰이 만료됐을 때 현재 사용자를 다시 인증하는 흐름이다. 토큰 추출 과정은 4번과 동일하고, 마지막에 `signIn` 대신 `reauthenticate`를 호출한다.

```swift
func reauthenticateWithApple(authorization: ASAuthorization) async throws {
    if let appleIDCredential = authorization.credential as? ASAuthorizationAppleIDCredential {
        guard let nonce = currentNonce else {
            fatalError("Invalid state: A login callback was received, but no login request was sent.")
        }
        guard let appleIDToken = appleIDCredential.identityToken else {
            print("Unable to fetch identity token")
            return
        }
        guard let idTokenString = String(data: appleIDToken, encoding: .utf8) else {
            print("Unable to serialize token string from data: \(appleIDToken.debugDescription)")
            return
        }

        let credential = OAuthProvider.appleCredential(
            withIDToken: idTokenString,
            rawNonce: nonce,
            fullName: appleIDCredential.fullName
        )

        try await Auth.auth().currentUser?.reauthenticate(with: credential)
    }
}
```

`fullName`을 `nil`로 넘기는 이유는 Apple이 최초 로그인 시에만 이름을 제공하기 때문이다. 재인증 시점엔 이미 저장된 값을 쓰면 된다.

#### 6. 토큰 취소 (Token Revocation)

Apple 로그인 계정을 탈퇴할 때 발급된 토큰을 취소하는 흐름이다.

Docs 샘플 코드는 `CryptoUtils.randomNonceString()`을 `try`로 호출하는 구조인데, 우리는 `randomNonceString()`을 `AppleLoginService` 안에 private 메서드로 직접 구현했고 실패 시 `fatalError`로 처리하기 때문에 `throws`가 아니다. 그래서 `do-catch`와 `try` 없이 바로 호출한다.

```swift
func deleteCurrentUser() {
    let nonce = randomNonceString()
    currentNonce = nonce
    let appleIDProvider = ASAuthorizationAppleIDProvider()
    let request = appleIDProvider.createRequest()
    request.requestedScopes = [.fullName, .email]
    request.nonce = sha256(nonce)

    let authorizationController = ASAuthorizationController(authorizationRequests: [request])
    authorizationController.delegate = self
    authorizationController.presentationContextProvider = self
    authorizationController.performRequests()
}
```

흐름은 3번 request 설정과 동일하다. 차이점은 인증 완료 후 delegate에서 `signIn` 대신 `revokeToken`을 호출한다는 것이다.

문제는 delegate 메서드가 하나인데 로그인과 토큰 취소 두 가지 경우를 모두 처리해야 한다는 점이다. 어떤 액션으로 Apple 인증을 요청했는지 delegate 안에서는 알 수 없기 때문에, 현재 액션 상태를 따로 들고 있다가 분기하는 구조가 필요하다.

#### 7. delegate 분기 처리

delegate 메서드가 하나이기 때문에 로그인과 토큰 취소를 구분할 방법이 필요하다. `AppleSignInAction` enum을 만들어서 현재 액션 상태를 들고 있다가 delegate 안에서 분기한다.

```swift
private enum AppleSignInAction {
    case signIn
    case delete
}
private var currentAction: AppleSignInAction = .signIn
```

`startSignInWithAppleFlow()` 호출 시 `.signIn`, `deleteCurrentUser()` 호출 시 `.delete`로 세팅한다. delegate에서는 `currentAction`으로 분기해서 각각 다른 처리를 한다.

```swift
extension AppleLoginService: ASAuthorizationControllerDelegate {
    func authorizationController(controller: ASAuthorizationController, didCompleteWithAuthorization authorization: ASAuthorization) {
        switch currentAction {
        case .signIn:
            guard let appleIDCredential = authorization.credential as? ASAuthorizationAppleIDCredential else {
                print("Unable to retrieve AppleIDCredential")
                return
            }
            guard let nonce = currentNonce else {
                fatalError("Invalid state: A login callback was received, but no login request was sent.")
            }
            guard let appleIDToken = appleIDCredential.identityToken else {
                print("Unable to fetch identity token")
                return
            }
            guard let idTokenString = String(data: appleIDToken, encoding: .utf8) else {
                print("Unable to serialize token string from data: \(appleIDToken.debugDescription)")
                return
            }
            let credential = OAuthProvider.appleCredential(
                withIDToken: idTokenString,
                rawNonce: nonce,
                fullName: appleIDCredential.fullName
            )
            Task {
                do {
                    let authResult = try await Auth.auth().signIn(with: credential)
                    print("Firebase 로그인 성공: \(authResult.user.uid)")
                } catch {
                    print("Firebase 로그인 실패: \(error.localizedDescription)")
                }
            }

        case .delete:
            guard let appleIDCredential = authorization.credential as? ASAuthorizationAppleIDCredential else {
                print("Unable to retrieve AppleIDCredential")
                return
            }
            guard let _ = currentNonce else {
                fatalError("Invalid state: A login callback was received, but no login request was sent.")
            }
            guard let appleAuthCode = appleIDCredential.authorizationCode else {
                print("Unable to fetch authorization code")
                return
            }
            guard let authCodeString = String(data: appleAuthCode, encoding: .utf8) else {
                print("Unable to serialize auth code string from data: \(appleAuthCode.debugDescription)")
                return
            }
            Task {
                do {
                    try await Auth.auth().revokeToken(withAuthorizationCode: authCodeString)
                    try await Auth.auth().currentUser?.delete()
                    print("계정 삭제 완료")
                } catch {
                    print("계정 삭제 실패: \(error.localizedDescription)")
                }
            }
        }
    }

    func authorizationController(controller: ASAuthorizationController, didCompleteWithError error: Error) {
        print("Sign in with Apple errored: \(error)")
    }
}
```

`user?.delete()` 대신 `Auth.auth().currentUser?.delete()`로 바꿨다. `private var user: User?`를 따로 들고 있을 필요 없이 Firebase에서 현재 유저를 직접 참조하면 되기 때문이다.

이후 채팅이나 다른 기능에서도 현재 로그인한 유저가 필요할 때는 `Auth.auth().currentUser`를 그대로 참조하는 방식으로 통일한다. Firebase Auth가 내부적으로 상태를 들고 있기 때문에 별도 프로퍼티로 유저를 관리할 필요가 없기 때문이다.

#### 8. 버튼 연동하기

`SignInWithAppleButton`은 두 가지 방식으로 쓸 수 있다. 버튼이 직접 `request`를 받아서 처리하는 방식과, 버튼은 트리거만 하고 실제 흐름을 외부에 맡기는 방식이다.

우리는 `ASAuthorizationController`를 `AppleLoginService` 안에서 직접 만들고 delegate도 거기서 처리하기 때문에 버튼의 `request`와 충돌이 생긴다. 그래서 버튼은 트리거 역할만 하고 실제 흐름은 `startSignInWithAppleFlow()`에 맡긴다.

```swift
SignInWithAppleButton(.signIn) { _ in } onCompletion: { _ in
    appleLoginService.startSignInWithAppleFlow()
}
```

`AppleLoginService`는 View에서 관찰할 내부 상태가 없기 때문에 `@Observable` 없이 일반 클래스로 두고 `@State`로 인스턴스를 만든다.

```swift
@State private var appleLoginService = AppleLoginService()
```

첫 번째 인자로 넘기는 레이블은 `.signIn`과 `.continue` 중 선택할 수 있다. 일반적으로는 `.signIn`을 많이 쓴다. `.continue`는 온보딩 중간 단계처럼 로그인이 선행되어야 다음으로 넘어갈 수 있는 흐름에서 쓰는 게 자연스럽다. JustChat은 `.signIn`으로 했다.

#### 10. 문제 수정

구현을 하면서 몇 가지 문제가 발견됐다.

- `SignInWithAppleButton`과 `startSignInWithAppleFlow()`를 동시에 호출해서 인증 창이 두 번 뜨는 문제
- 버튼의 `request` 클로저를 무시해서 `authorizedScopes`가 비어있는 문제
- `currentAction`을 세팅하는 부분이 없어서 delegate 분기가 실제로 동작하지 않는 문제
- `SignInWithAppleButton`을 쓰면 nonce 흐름이 꼬이는 구조적 문제

---

##### 1. 인증 창 두 번 뜨는 문제 & authorizedScopes 비어있는 문제

`SignInWithAppleButton`은 자체적으로 Apple 인증 흐름을 실행하고 결과를 `onCompletion`으로 돌려준다. 그런데 `onCompletion`에서 또 `startSignInWithAppleFlow()`를 호출하니까 `ASAuthorizationController`가 한 번 더 실행되면서 인증 창이 두 번 뜬 것이다.

`authorizedScopes`가 비어있던 것도 같은 맥락이다. 버튼의 `request` 클로저를 `{ _ in }`으로 무시했기 때문에 `.fullName`, `.email` scope가 Apple에 전달되지 않았다. 이름/이메일 선택 화면이 뜨지 않았던 이유도 이거였다.

```swift
SignInWithAppleButton(.signIn) { _ in } onCompletion: { _ in
    appleLoginService.startSignInWithAppleFlow()
}
```

##### 2. currentAction 미연결 문제

```swift
    // 생략
}
```

##### 4. 해결

###### SessionStore

로그인 상태를 앱 전체에서 관리하기 위해 `SessionStore`를 만들었다. `AuthState` enum으로 4가지 상태를 정의한다.

```swift
enum AuthState: Equatable {
    case checking          
    case signedIn(AppUser)
    case guest
    case signedOut
}

@Observable
final class SessionStore {
    var state: AuthState = .checking
    var showsLoginPrompt = false
    var loginPromptMessage = ""
    
    private var authListener: AuthStateDidChangeListenerHandle?

    var isGuest: Bool {
        if case .guest = state { return true }
        return false
    }

    var currentUser: AppUser? {
        if case .signedIn(let user) = state { return user }
        return nil
    }

    func checkExistingSession() {
        authListener = Auth.auth().addStateDidChangeListener { [weak self] _, user in
            guard let self else { return }
            if let user {
                self.state = .signedIn(AppUser(from: user))
            } else {
                self.state = .signedOut
            }
        }
    }

    func continueAsGuest() {
        state = .guest
    }

    func signOut() {
        state = .signedOut
    }

    @discardableResult
    func requireLogin(message: String = "로그인이 필요한 기능입니다.") -> Bool {
        guard isGuest else { return true }
        loginPromptMessage = message
        showsLoginPrompt = true
        return false
    }
    
    deinit {
        if let authListener {
            Auth.auth().removeStateDidChangeListener(authListener)
        }
    }
}
```

- `.checking` - 앱 실행 직후 기존 세션 확인 중
- `.signedIn(AppUser)` - 로그인 완료
- `.guest` - 게스트 모드
- `.signedOut` - 로그아웃 상태

`@discardableResult`는 반환값을 무시해도 컴파일 경고가 뜨지 않도록 하는 Attribute이다. 처음에 AI를 통해 Mock UI를 만들 때 이미 구현이 되어있던 부분인데, `requireLogin()`은 `Bool`을 반환하지만 호출하는 쪽에서 반환값이 필요 없는 경우도 있기 때문에 붙여둔 것이다. 예를 들어 게스트가 메시지 전송을 시도했을 때 로그인 유도 알림만 띄우고 반환값은 신경 쓰지 않아도 되는 상황이다.

`addStateDidChangeListener`는 `AuthStateDidChangeListenerHandle`을 반환하는데, 이를 저장해두지 않으면 경고가 뜨고 나중에 리스너를 해제할 수도 없다. `authListener`에 저장해두고 `deinit`에서 `removeStateDidChangeListener`로 해제한다.

Firebase Auth는 `addStateDidChangeListener`로 로그인 상태 변화를 감지할 수 있다. `AppleLoginService`에서 Firebase signIn이 성공하는 순간 Auth 상태가 바뀌고, 리스너가 이를 감지해서 `SessionStore`의 `state`를 업데이트하는 구조다. View는 `state`만 바라보면 되기 때문에 로그인 성공 후 화면 전환을 별도로 처리할 필요가 없다.

###### 모델링

Firebase Auth 연동을 위해 앱에서 사용할 유저 모델을 정의했다. 기존에 프리뷰용으로 만들어뒀던 `AppUser`를 교체했고, 프리뷰용 샘플 데이터도 이 시점에 걷어냈다.

2년 전에는 로그인 성공 후 Firebase Auth의 값을 꺼내서 모델에 하나씩 대입하고 DB에 저장하는 방식이었다.

```swift
// SignViewModel.swift (2024)
let model = UserModel(uid: user.uid, email: email, isBlock: false, nickName: "", profileImageUrl: "https://...", isAgree: false)
self?.signManager.saveUserData(user: model)
```

이번엔 `init(from:)`을 정의해서 `FirebaseAuth.User`를 넘기면 바로 `AppUser`로 변환되도록 했다. `addStateDidChangeListener`에서 Auth 상태가 바뀔 때마다 DB 조회 없이 바로 `AppUser`를 만들 수 있기 때문이다.

```swift
import FirebaseAuth

struct AppUser: Identifiable, Equatable {
    let id: String
    let displayName: String
    let profileImageURL: URL?
    
    init(from user: FirebaseAuth.User) {
        self.id = user.uid
        self.displayName = user.displayName ?? "사용자"
        self.profileImageURL = user.photoURL
    }
}
```

- `id` - Firebase Auth의 `uid`. 유저를 식별하는 고유값이다.
- `displayName` - 소셜 로그인 시 제공되는 이름. 없으면 "사용자"로 대체한다.
- `profileImageURL` - 소셜 로그인 시 제공되는 프로필 이미지 URL.

###### 커스텀 버튼

`SignInWithAppleButton`을 걷어내고 Apple 가이드라인을 지키면서 커스텀 버튼으로 교체했다. 처음엔 SF Symbols이나 에셋을 직접 추가하는 방법을 생각했는데, AI를 통해 `ASAuthorizationAppleIDButton`을 `UIViewRepresentable`로 감싸서 외형만 가져오는 방법이 있다는 걸 알게 됐다.

`ASAuthorizationAppleIDButton`을 `UIViewRepresentable`로 감싸면 버튼 외형은 그대로 유지하면서 탭 동작은 `.onTapGesture`로 직접 제어할 수 있다.

```swift
struct AppleSignInButton: UIViewRepresentable {
    func makeUIView(context: Context) -> ASAuthorizationAppleIDButton {
        return ASAuthorizationAppleIDButton(type: .signIn, style: .black)
    }
    
    func updateUIView(_ uiView: ASAuthorizationAppleIDButton, context: Context) {}
}
```

View에서는 `.onTapGesture`로 `startSignInWithAppleFlow()`만 트리거한다.

```swift
AppleSignInButton()
    .frame(height: 52)
    .clipShape(RoundedRectangle(cornerRadius: AppTheme.cornerRadius))
    .onTapGesture {
        appleLoginService.startSignInWithAppleFlow()
    }
```

`UIViewRepresentable`은 UIKit의 뷰를 SwiftUI에서 사용할 수 있도록 감싸주는 프로토콜이다. SwiftUI에서 제공하지 않는 UIKit 컴포넌트를 쓸 때 활용한다. 자세한 내용은 [uiviewrepresentable Docs](https://developer.apple.com/documentation/swiftui/uiviewrepresentable){:target="_blank"}를 참고.

`makeUIView(context:)`에서 UIKit 뷰를 생성하고, `updateUIView(_:context:)`에서 SwiftUI 상태 변화에 따라 뷰를 업데이트한다. 이번엔 버튼 외형만 필요하고 상태 업데이트가 없어서 `updateUIView`는 비워뒀다.

현재 `currentAction`의 기본값이 `.signIn`으로 선언만 된 상태라 `deleteCurrentUser()`를 호출해도 delegate에서 항상 `.signIn` 분기를 타게 된다. 각 함수 시작 시점에 `currentAction`을 세팅해줘야 한다.

###### 화면 전환 연결하기

2년 전에는 `SceneDelegate`에서 `GreetingViewController`에 클로저를 주입하고, 로그인 성공 시 `window?.rootViewController`를 직접 교체하는 방식으로 화면을 전환했다.

```swift
// SceneDelegate.swift (2024)
greetingVC = GreetingViewController(
    appleTapped: { [weak signViewModel] in
        signViewModel?.appleLoginDidTapped()
    },
    guestTapped: {
        self.customUser = CustomUser(guestUID: "guest")
        self.switchToMainTabBarController()
    },
    viewModel: signViewModel)
self.window?.rootViewController = tabbarController
```

이번엔 `RootView`가 `SessionStore`의 `state`를 바라보고 알아서 화면을 전환한다. `LoginView`에서 클로저 파라미터도 걷어내고, `@Environment`로 `session`을 받아서 직접 호출하는 구조로 정리했다.

```swift
// RootView.swift
struct RootView: View {
    @State private var session = SessionStore()

    var body: some View {
        Group {
            switch session.state {
            case .checking:
                ProgressView()
            case .signedOut:
                LoginView()
            case .signedIn, .guest:
                MainTabView()
            }
        }
        .environment(session)
        .task {
            session.checkExistingSession()
        }
    }
}
```

`.task`에서 `checkExistingSession()`을 호출해서 Auth 리스너를 붙인다. 이후 `AppleLoginService`에서 Firebase signIn이 성공하면 Auth 상태가 바뀌고, 리스너가 감지해서 `state`가 `.signedIn`으로 업데이트되면서 자동으로 `MainTabView`로 전환된다. 화면 전환 코드를 별도로 작성할 필요가 없다.
