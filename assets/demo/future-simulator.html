왜 Future를 쓰는가?

completion handler와 Future — 같은 비동기 작업, 다른 결과

Callback 비동기 결과를 클로저 안으로 전달한다. 값이 클로저 내부에 갇혀 밖에서 Combine 연산을 붙일 수 없다.
Future 비동기 결과를 Publisher로 반환한다. 값이 스트림으로 흘러나와 밖에서 .map, .decode 등 Combine 파이프라인을 자유롭게 연결할 수 있다.

Completion Handler

func loadFile( completion: @escaping (Result<Data, Error>) -> Void ) { guard let url = ... else { completion(.failure(err)) return } completion(.success(data)) } // 호출 loadFile { result in // 값이 여기 갇힘 // 밖에서 .map 불가 }
Combine 파이프라인
.map { decode($0) } ← 불가
.filter { ... } ← 불가
.sink { } ← 불가

Future

func loadFile() -> AnyPublisher<Data, Error> { Future { promise in guard let url = ... else { promise(.failure(err)) return } promise(.success(data)) } .eraseToAnyPublisher() } // 호출 — Publisher 반환 loadFile() // ← 값이 밖으로 흘러나옴
Combine 파이프라인
.map { decode($0) }
.filter { $0.isValid }
.sink { updateUI($0) }

결론
completion handler는 결과를 클로저 안에 가둔다. 그래서 Combine 파이프라인과 연결할 수 없다.
Future는 결과를 Publisher로 반환한다. 그래서 .map, .filter, .sink 등 Combine 연산을 자유롭게 체이닝할 수 있다.

Future는 "기존 콜백 코드를 Combine 파이프라인의 시작점으로 만드는 진입점"이다.
